## The 19 Features Used for Training

### Why exactly these 19? Why not use ALL columns?

The dataset has more columns than features. Three are **deliberately excluded**:

| Excluded Column | Why |
|-----------------|-----|
| `scheduled_class` | This IS the timetable. Using it = feeding the answer into the question. |
| `prev_hour_occupied` | Directly derived from timetable → data leakage |
| `next_hour_occupied` | Same problem — the model would cheat, not learn |

> **Data Leakage in plain English:** Imagine a student taking an exam where the teacher accidentally includes the answer key in the question paper. The student scores 100% — but has learned nothing. In production, the model would fail because it can't access next semester's timetable. We train it on *environmental signals* only.

### The 19 Features That ARE Used

```python
FEATURE_COLS = [
    # TEMPORAL (5 features)
    "day_of_week",      # 0=Mon … 5=Sat
    "hour",             # 8 to 20
    "hour_sin",         # sin(2π·hour/24) — cyclical time encoding
    "hour_cos",         # cos(2π·hour/24) — cyclical time encoding
    "is_weekday",       # 1 if Mon–Fri, 0 if Sat

    # ROOM STATIC (3 features)
    "floor",            # Which floor the room is on
    "is_lab",           # 1 if it's a computer/lab room
    "is_special",       # 1 if it's locked/seminar/special use

    # RATE-BASED (4 features — the "smart" features)
    "room_overall_occupancy_rate",   # How often this room is busy in general
    "block_hour_occupancy_rate",     # How busy the entire block is at this hour
    "neighbor_hour_occupancy_rate",  # How busy adjacent rooms are right now
    "room_popularity_bucket",        # Quartile rank: 0 (quiet) to 3 (always busy)

    # BLOCK ONE-HOT (7 features)
    "block_A", "block_B", "block_C", "block_D",
    "block_E", "block_F", "block_L",
]
```

## Step 1: Loading Data & Splitting

```python
df = pd.read_csv(DATASET_CSV)
X  = df[FEATURE_COLS].values   # Shape: (n_rows, 19)
y  = df[TARGET_COL].values     # Shape: (n_rows,) — 0 or 1

X_train, X_test, y_train, y_test = train_test_split(
    X, y,
    test_size=0.2,          # 80% train, 20% test
    random_state=42,        # Reproducibility — always the same split
    stratify=y              # ← Key: keeps class ratio balanced
)
```

### What does `stratify=y` actually do?

**Without stratify (random):**
- Training set: 85% not-usable, 15% usable ← biased!
- Test set: 60% not-usable, 40% usable ← different distribution!

**With `stratify=y`:**
- Both train and test sets have the **exact same ratio** as the full dataset
- Prevents the model from being evaluated on an unrepresentative sample

## Step 2: Model 1 — Logistic Regression (Baseline)

```python
lr_pipe = Pipeline([
    ("scaler", StandardScaler()),
    ("clf",    LogisticRegression(max_iter=1000, random_state=42)),
])
lr_pipe.fit(X_train, y_train)
```

### What is a Pipeline?
A Pipeline chains steps together. When you call `.fit()`, it:
1. Fits the scaler on training data → transforms it
2. Passes transformed data into the classifier → trains it

When you call `.predict()` on new data, it automatically applies the same scaler first. This is crucial — if you forgot to scale test data, predictions would be garbage.

### Why StandardScaler?
Logistic Regression uses gradient descent. Features with large ranges (like `hour`: 8–20) would dominate over small-range features (like `is_lab`: 0–1) without scaling.

```
Before scaling:    hour=14, is_lab=1, room_rate=0.42
After scaling:     hour=0.31, is_lab=1.73, room_rate=-0.61
                   (all features now on same scale: mean=0, std=1)
```

### What Logistic Regression actually does:
It learns a weight for each feature:
```
P(usable) = sigmoid(
    w1*day_of_week + w2*hour + w3*hour_sin + ... + bias
)
```
If P > 0.5 → predict Usable. If P ≤ 0.5 → predict Not Usable.

**Limitation:** It can only draw a *straight line* between classes. The real boundary is not linear (e.g., "usable at 10 AM *only if* the next hour is also free"), so this model underperforms.

## Step 3: Model 2 — Random Forest

```python
rf = RandomForestClassifier(
    n_estimators=100,   # 100 decision trees
    max_depth=15,       # Max tree depth — prevents overfitting
    random_state=42,
    n_jobs=-1           # Use all CPU cores
)
rf.fit(X_train, y_train)
```

### No scaler needed! Why?
Decision trees split on feature thresholds (`hour > 12?`). They don't care about the absolute scale — the threshold adapts automatically.

### How 100 trees become one answer:

```
Tree 1: is_special=0 AND hour<18 AND block_rate<0.6  → USABLE
Tree 2: is_lab=0 AND pop_bucket<3 AND is_weekday=1   → NOT USABLE
Tree 3: floor>0 AND room_rate<0.5 AND hour>9         → USABLE
...
Tree 100: ...                                         → USABLE

Vote: 67 say USABLE, 33 say NOT USABLE → Final: USABLE ✓
```

Each tree is trained on a **random subset** of rows and features. This "randomness" prevents all trees from making the same mistake — errors cancel out.

### Feature Importance Output
```

neighbor_hour_occupancy_rate 0.1102  ##################
```

## Step 4: Model 3 — Neural Network (ANN / MLP)

```python
ann_pipe = Pipeline([
    ("scaler", StandardScaler()),
    ("mlp", MLPClassifier(
        hidden_layer_sizes=(64, 32),    # Two hidden layers
        max_iter=500,
        early_stopping=True,
        validation_fraction=0.1,        # 10% of train set = validation
        n_iter_no_change=20,            # Stop if no progress for 20 epochs
    )),
])
```

### Network Architecture:

```
INPUT LAYER          HIDDEN LAYER 1      HIDDEN LAYER 2      OUTPUT
[19 features]   →   [64 neurons]    →   [32 neurons]    →   [1 neuron]
                      ReLU                ReLU                Sigmoid
                                                           (0=not usable,
                                                            1=usable)
```

### What each layer learns (conceptually):
- **Layer 1 (64 neurons):** Learns basic patterns — "busy hours," "popular blocks"
- **Layer 2 (32 neurons):** Combines those patterns — "this is a quiet room during a busy period"
- **Output:** Converts the combined signal into a probability: P(usable)

### Early Stopping in action:
```
Epoch 1:   validation_loss = 0.621
Epoch 50:  validation_loss = 0.312  ← improving
Epoch 100: validation_loss = 0.298  ← improving
Epoch 120: validation_loss = 0.299  ← no improvement
Epoch 121: validation_loss = 0.300  ← no improvement
...
Epoch 140: validation_loss = 0.301  ← 20 epochs no improvement → STOP!
```
This prevents the model from memorising training data (overfitting).

## Step 5: Comparison & Winner Selection

```python
best_name = max(results, key=lambda k: results[k]["f1"])
```

### Why F1-macro and not Accuracy?

**Scenario:** Dataset has 80% "Not Usable" rows.
- A dumb model that always predicts "Not Usable" → **80% accuracy** ← looks great!
- But its F1 for the "Usable" class → **0.0** ← exposed!

F1 balances **Precision** and **Recall** for both classes equally.

### Metrics Explained:

| Metric | Formula | What it means |
|--------|---------|---------------|
| **Precision** | TP / (TP + FP) | Of rooms I said were usable, how many actually were? |
| **Recall** | TP / (TP + FN) | Of all actually usable rooms, how many did I find? |
| **F1** | 2 × (P × R) / (P + R) | Harmonic mean — penalises imbalance between P and R |
| **ROC-AUC** | Area under ROC curve | Overall discriminative ability (0.5 = random, 1.0 = perfect) |

### Confusion Matrix breakdown:
```
                    Predicted: Not Usable    Predicted: Usable
Actual: Not Usable       TN ✓                    FP ✗ (false alarm)
Actual: Usable           FN ✗ (missed!)           TP ✓
```
- **FP** = App tells student "room is free" but it's busy → **worst outcome** (bad UX)
- **FN** = App says "not usable" but it actually was free → missed opportunity (less harmful)

## Step 6: Saving the Best Model

```python
with open(MODEL_PKL, "wb") as f:
    pickle.dump({
        "model":        best_model,     # The full trained pipeline/model object
        "model_name":   best_name,      # e.g., "Random Forest"
        "feature_cols": FEATURE_COLS,   # The exact ordered list of 19 features
    }, f)
```

### Why save `feature_cols` alongside the model?
When the live app makes a prediction, it builds a feature vector from real-time data. The features **must be in the exact same order** as during training. If they're shuffled, the model receives garbage input and silently produces wrong predictions.

### How the live app uses it:
```python
# Loading the model
with open("Models/room_usability_model.pkl", "rb") as f:
    bundle = pickle.load(f)

model        = bundle["model"]
feature_cols = bundle["feature_cols"]

# Predicting for a real query
X_new = build_features_for_room(room, day, hour)  # returns dict
X_vec = [X_new[col] for col in feature_cols]      # ordered correctly!

prediction   = model.predict([X_vec])[0]  # 0 or 1
probability  = model.predict_proba([X_vec])[0][1]  # 0.0 to 1.0
```

## Known Failure Cases (from the script's own analysis)

### Case 1: Logistic Regression underperforms
The "usable" decision is inherently non-linear. A room that is free at 9 AM is usable *only if* the next hour is also free. That "AND" logic cannot be captured by a straight decision boundary without polynomial feature expansion.

### Case 2: B-Block is always "Not Usable"
CRMG and Embedded Lab are in `SPECIAL_ROOMS`. All B-Block slots are labelled `usable=0`. The model correctly learns this — but provides no value for B-Block navigation. **Real-world fix:** Integrate a room booking calendar API for special rooms.
